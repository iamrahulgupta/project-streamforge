Architecture Overview

StreamForge is a distributed event streaming platform composed of several key components working together:

System Components

🦀 Broker (Rust)

Core messaging engine with cluster coordination, replication, and partition management.

💾 Storage (C++)

High-performance log engine with memory-mapped file I/O and segment management.

📦 SDK (Go)

Client library for producers and consumers with batching and offset management.

⚙️ Admin (Go)

CLI and REST API for cluster management and monitoring.

Broker Architecture

Cluster Management

Manages broker membership and coordination within the cluster.

  • Membership: Tracks active brokers and their health status
  • Heartbeat: Periodic health checks every 3-5 seconds
  • Metadata: Maintains topic and partition metadata
  • Leader Election: Raft-based consensus for cluster coordination

Replication

Manages data replication across brokers for fault tolerance.

  • Leader: Manages replication to follower brokers
  • Follower: Syncs messages from leader replica
  • LogSync: Maintains consistency using append entries protocol
  • In-Sync Replicas (ISR): Tracks replicas that are caught up with leader

Partition Management

Handles message storage and retrieval at the partition level.

  • Manager: Stores and retrieves messages efficiently
  • Offset Manager: Tracks consumer group offsets
  • Commit Log: Persists offset commits for durability

Network Protocol

Implements inter-broker and client communication.

  • gRPC Server: Accepts client and broker connections
  • gRPC Client: Makes connections to other brokers
  • Protocol Buffers: Type-safe message serialization
  • Connection Pooling: Reuses connections for performance

Storage Architecture

Log Engine

Main storage component managing the complete message lifecycle.

LogEngine
├── Segment Files
│   ├── Segment 1 (0-999999 offsets)
│   ├── Segment 2 (1000000-1999999 offsets)
│   └── Segment N (...)
├── Index Files
│   ├── Index 1 (B-tree for Segment 1)
│   └── Index N (...)
└── Commit Log
    └── Consumer group offsets

Segment Management

  • Fixed Size: Segments rotate when reaching size limit (default 1GB)
  • Automatic Rotation: Old segments can be deleted based on retention
  • Fast Lookups: Index files enable O(log n) message retrieval
  • Memory-Mapped I/O: Direct kernel page cache access

Consistency Model

Write Consistency

  • Leader Writes: Always written to leader partition
  • Replication: Async replication to follower brokers
  • Min In-Sync Replicas (min.insync.replicas): Durability guarantee
  • ACK Modes:
    • acks=0: No acknowledgment (fastest, least safe)
    • acks=1: Leader acknowledgment
    • acks=-1: All in-sync replicas acknowledgment (slowest, safest)

Read Consistency

  • Read from Leader: Always reads from partition leader
  • High Water Mark: Maximum offset available to consumers
  • Offset Commits: Consumer group offset persistence
  • Ordering: Messages in partition maintain order per key

Performance Optimizations

Memory-Mapped Files

Zero-copy reads by mapping file directly to process memory:

  • Direct kernel page cache access
  • Automatic OS caching management
  • Minimal data copying
  • High throughput for sequential reads

Batch Processing

Accumulates multiple messages before network transmission:

  • Reduced network roundtrips
  • Higher throughput with similar latency
  • Configurable batch size and timeout

Async I/O

Non-blocking operations throughout the stack:

  • Tokio async runtime for Rust broker
  • Go goroutines for concurrent client connections
  • No thread-per-connection overhead

Connection Pooling

Reuses connections to reduce overhead:

  • TCP connection pooling to brokers
  • Connection keepalive and health checks
  • Automatic reconnection on failure

Failure Handling

Broker Failures

  • Detection: Heartbeat timeout (3-5s)
  • Leader Election: Raft-based election triggered
  • Partitions: Automatically led by other broker
  • Client Failover: SDK retries on different broker

Network Failures

  • Client Retry: Exponential backoff up to max retries
  • Broker Discovery: Clients fetch metadata for all brokers
  • Rack Awareness: Replicas spread across different racks

Storage Failures

  • Replication: Data available on other replicas
  • Durability: fsync() with configurable interval
  • Checksums: Verify message integrity

Scaling Architecture

Vertical Scaling

  • Increase partition count per broker
  • Larger batch sizes for higher throughput
  • More CPU cores for connection handling

Horizontal Scaling

  • Add more brokers to cluster
  • Reassign partitions across brokers
  • Scale producers and consumers independently
