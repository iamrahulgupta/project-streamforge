Performance Tuning

Broker Optimization

Memory Management:

  • Allocate 4GB minimum for production deployments
  • Increase to 8-16GB for high-throughput scenarios
  • Monitor GC (garbage collection) in logs
  • Enable page cache (kernel will manage automatically)

Network Optimization:

  • Increase socket buffer sizes: SO_SNDBUF=2097152, SO_RCVBUF=2097152
  • Enable TCP_NODELAY for low-latency workloads
  • Set TCP window scaling: net.ipv4.tcp_window_scaling=1
  • Increase connection backlog: net.core.somaxconn=16384

Storage Optimization:

  • Use SSD for log directories (10x faster than HDD)
  • Enable compression for high-volume topics (50% space reduction)
  • Set segment size to 1GB (balance between rotation and indexing)
  • Configure retention based on storage capacity

Producer Tuning

// For high throughput
const config = {
  batchSize: 100000,           // Batch 100K messages
  batchTimeoutMs: 1000,        // Or 1s timeout
  compression: 'snappy',       // Enable compression
  maxInFlight: 10,             // Parallel batches
  requiredAcks: 1,             // Leader ACK only
};

// For low latency
const config = {
  batchSize: 10,               // Small batches
  batchTimeoutMs: 1,           // Immediate sends
  compression: 'none',         // No compression
  maxInFlight: 1,              // Sequential
  requiredAcks: 1,             // Leader ACK
};

// Balanced configuration
const config = {
  batchSize: 1000,             // 1K message batches
  batchTimeoutMs: 100,         // 100ms timeout
  compression: 'lz4',          // Fast compression
  maxInFlight: 5,              // Moderate parallelism
  requiredAcks: 1,             // Default safety
};

Consumer Tuning

// For high throughput
const config = {
  maxPollRecords: 5000,        // Large batches
  fetchMaxWaitMs: 5000,        // Wait longer
  sessionTimeoutMs: 30000,     // Generous timeout
  autoCommit: true,            // Automatic offset commit
};

// For low latency
const config = {
  maxPollRecords: 100,         // Small batches
  fetchMaxWaitMs: 100,         // Poll frequently
  sessionTimeoutMs: 6000,      // Quick failure detection
  autoCommit: false,           // Manual commit
};

// Exactly-once semantics
const config = {
  maxPollRecords: 500,         // Batch process
  fetchMaxWaitMs: 500,         // Regular polling
  sessionTimeoutMs: 10000,     // Standard timeout
  autoCommit: false,           // ExactlyOnceProcessor
};

Hardware Recommendations

Single Broker (Development/Testing)

  • CPU: 2-4 cores @ 2.4GHz+
  • Memory: 4GB (2GB heap, 2GB OS/page cache)
  • Storage: 20GB SSD
  • Network: 1Gbps
  • Disk I/O: 500+ IOPS

3-Broker Cluster (Production Low Volume)

  • CPU: 8 cores @ 2.6GHz+ per broker
  • Memory: 8-12GB per broker (4GB heap, 4-8GB page cache)
  • Storage: 256GB SSD per broker (RAID-10 recommended)
  • Network: 10Gbps (or 2x 5Gbps)
  • Disk I/O: 2000+ IOPS per broker

Large Cluster (Production High Volume)

  • CPU: 16-32 cores @ 2.8GHz+ per broker
  • Memory: 64-128GB per broker (32GB heap, 32-96GB page cache)
  • Storage: 2-4TB NVMe SSD per broker (RAID-10)
  • Network: 100Gbps (or multiple 10Gbps bonds)
  • Disk I/O: 10000+ IOPS per broker

Capacity Planning

Retention Calculation

Required Storage = (Messages/Second × Message Size × Retention Seconds)

Example:
- Throughput: 100,000 msg/s
- Average message: 10KB
- Retention: 7 days (604,800 seconds)

Storage = (100,000 × 10KB × 604,800) = 604.8TB
Per Broker (3 replicas): 201.6TB
With compression (40%): 120.96TB

Partition count recommendation:
- Total output throughput / max broker throughput
- 100,000 msg/s / 850,000 msg/s ≈ 1 partition minimum
- Use 6-12 partitions for better parallelism

Network Bandwidth

Required Bandwidth = (Throughput × Message Size × Replication Factor)

Example:
- Throughput: 100,000 msg/s (100 MB/s input)
- Message size: 10KB
- Replication factor: 3

Network = 100 MB/s × 3 = 300 MB/s output
With acks=all: 400-500 MB/s peak
Required: 1Gbps minimum, 10Gbps recommended for production

Scaling Strategies

Vertical Scaling (Increase broker resources)

  • Benefits: Simple to implement, lower operational overhead
  • Limits: Hardware constraints, single point of failure
  • When to use: Latency-sensitive workloads, small deployments
  • Process: Add CPU/Memory → Restart broker → Reassign partitions

Horizontal Scaling (Add more brokers)

  • Benefits: Nearly unlimited capacity, high availability
  • Limits: Network coordination overhead, rebalancing time
  • When to use: Throughput-driven workloads, large deployments
  • Process: Add broker → Assign partitions → Rebalance

Partition Scaling

  • Increase partitions: Parallel consumer processing, higher throughput
  • Decrease partitions: Reduced operational complexity
  • Rebalancing time: 1 minute per 10GB data per partition
  • Recommendation: Start with 3N partitions (N = broker count)

Common Performance Issues

High Latency

Symptoms: P99 latency > 50ms, producer timeouts

Causes & Solutions:

  • GC pauses → Increase heap size, tune GC settings
  • Disk I/O bottleneck → Use SSD, enable compression
  • Network congestion → Upgrade network, add brokers
  • Large batch sizes → Reduce batch size, reduce acks level

Low Throughput

Symptoms: Messages/sec below expected, lag building up

Causes & Solutions:

  • Batch size too small → Increase to 1000+
  • Compression overhead → Use lz4 or snappy
  • Too few consumers → Add consumer groups
  • Not enough partitions → Add partitions, rebalance

High Memory Usage

Symptoms: Process using >90% of heap, GC overhead

Causes & Solutions:

  • Batch buffer bloat → Reduce max_in_flight
  • Cache too large → Reduce cache_size config
  • Message retention too long → Lower retention_ms
  • Many partitions → Check partition:broker ratio

Benchmarking Guide

Running Your Own Benchmarks

# Latency test (Go)
go test -run TestProducerLatency -bench=. -benchtime=10s

# Throughput test (Rust)
cargo bench --bench broker_throughput

# End-to-end test
./benchmark-e2e \
  --brokers localhost:9092 \
  --messages 100000 \
  --message-size 10240 \
  --topics 10 \
  --partitions 50

Comparison Methodology

  • Use identical hardware for fair comparison
  • Warm up system before measuring (5 minutes)
  • Run tests multiple times, discard outliers
  • Measure under load (sustained throughput)
  • Test failure scenarios (broker restart, network)
